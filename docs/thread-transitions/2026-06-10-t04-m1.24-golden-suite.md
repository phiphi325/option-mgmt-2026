### What M1.24 added to the engine

- **`engine.decision.serialize.serialize_canonical(decision) -> dict`** — the
  canonical-JSON serializer for `DailyDecision`. Sorted keys, **6-decimal float
  rounding** (absorbs Apple-Silicon-vs-Linux ULP差), naive datetime → UTC ISO,
  enum → `.value`, tuple → list, frozenset → sorted-by-repr list, Pydantic via
  `model_dump(mode="json")`. Re-exported from `engine.decision`. **Test-infra
  only** — no production path imports it. This new public function is what
  drove the `1.6.0` → `1.7.0` minor bump (ADR-0005).

### Test infrastructure (`packages/engine/tests/`)

- `test_master_decision_goldens.py` — parametrized replay harness, one case
  per fixture; byte-compares `serialize_canonical(produce_daily_decision(**inputs))`
  against the committed `expected.json`.
- `test_master_decision_goldens_meta.py` — suite-level invariants: all 8 V1
  emit codes covered, both M1.11b collar paths, M1.12 escalation, all 6
  ADR-0002 regimes, `inputs_hash` stability, version stamps.
- `test_serialize_canonical.py` — unit tests for the serializer.
- `test_regenerate_decision_goldens_idempotent.py` — byte-idempotency guard.
- `_golden_helpers.py` — `load_inputs`, `canonical_dumps`, `list_fixtures`.
- `scripts/regenerate_decision_goldens.py` — humans run after schema changes.
- 12 fixtures under `tests/fixtures/master_decisions/` (one dir each:
  `README.md` + `inputs.json` + `expected.json`).

## Decisions made

- **Q1 — engine version bump for a test-infra function: YES.**
  `serialize_canonical` is a new public function under
  `packages/engine/engine/`, so strict ADR-0005 reading mandates a minor bump.
  `1.6.0` → `1.7.0`. (The dev spec's front matter originally said "No"; that
  was superseded — see the spec's post-ship reconciliation appendix.)
- **Q2 — float rounding at 6 decimal places.** Max precision loss 5e-7, well
  below the 2-decimal UI display. Held up across macOS + Linux (Q3 resolved in
  practice — no loosening needed).
- **Q5 — API-layer goldens deferred to M1.25.** Engine-layer goldens are
  sufficient to lock the shape for the frontend cycle.
- **Fixture 05 scope refinement.** The spec framed it as `LOW_IV_TREND +
  style=income overrides buy_put`, but `high_iv_sell_call` only fires for
  `{HIGH_IV_EVENT, HIGH_IV_PIN, LOW_IV_RANGE}` per `rules.yaml` — not
  `LOW_IV_TREND`. Repurposed to `LOW_IV_RANGE + income + iv_rank=55`. Directory
  slug kept stable to avoid rename churn; the fixture README documents it.

## Gotchas & non-obvious things

- **Fixture IV-scale bug (caught + fixed in PR #4).** The first-pass
  `inputs.json` files put `iv_rank` / `iv_percentile` / `iv_rank_change_1d` on a
  **`[0-100]`** scale (`iv_rank: 40.0`). The engine's canonical scale is
  **`[0-1]`** (`MarketStateResult` + `classify()`). All 12 fixtures were
  rescaled. **For future fixture authors:** `iv_rank`, `iv_percentile`,
  `trend_strength`, `breakout_signal`, `oi_concentration_at_max_pain` are all
  `[0-1]`; `iv_rank_change_1d` is the one-day delta in `[-1, 1]`.
- **Enum casing in fixtures.** `Bias` / `RecommendedAction` / `Regime` /
  `OptionType` use **UPPERCASE** wire values (`BEARISH`, `SELL_CALL_PARTIAL`,
  `HIGH_IV_EVENT`, `CALL`). `ProfileStyle` / `RiskTolerance` / `IncomeNeed` use
  **lowercase** (`income`, `moderate`, `medium`). Mismatches fail enum
  reconstruction in `load_inputs`.
- **`serialize_canonical` ruff/PEP-8 traps.** `from datetime import UTC` (not
  `timezone.utc`, UP017 under py314). `Settings` version imports: alias
  `__version__` lowercase (`_engine_version`, N812) but `DEFAULT_WEIGHTS`
  uppercase (`_DEFAULT_WEIGHTS`, N811) — mixed casing is intentional.
- **GitHub integration migrated Composio → MCP mid-thread.** Partway through,
  the Composio GitHub integration was disabled at the platform level (the
  Settings → Integrations card flagged it "replaced by a native/MCP
  integration or restricted by admin policy"). Commits via
  `GITHUB_COMMIT_MULTIPLE_FILES` stopped working with a "not available in this
  runtime" error. **Fix:** reconnect GitHub as the **MCP** integration at
  Settings → Integrations. The MCP action surface differs from Composio:
  multi-file commit is `github__push_files`; merge is `github__merge_pull_request`;
  PR detail/checks via `github__pull_request_read` (`method: get` /
  `get_check_runs`). The MCP server does **not** expose a workflow-log download
  or a job re-run action (Composio did) — re-running a failed CI job is a
  manual click in the Actions UI.
- **Smoke CI flake.** PR #3's smoke job failed once on
  `docker pull postgres:16-alpine` timing out against Docker Hub — an infra
  flake before any test ran, not a code issue. A re-run went green. Watch for
  this on future PRs; just re-run the job.
