## Decisions made

(none — this thread documents an already-shipped state; no new contracts)

The shipped impl's **deviations from the M1.11b dev spec** are recorded
canonically in the spec's new "Post-ship reconciliation" appendix and in the
`CHANGELOG [1.6.0]` "Plan deviations" subsection. Summary:

1. **`_emit_to_intent` was not extracted.** Spec proposed a string-intent
   helper; shipped impl inlines `if action.emit is EmittedAction.OPEN_COLLAR:`
   inside the per-action loop. The rule pipeline already emits typed
   `EmittedAction`, so an intermediate string layer adds no value.
2. **The spec draft listed `SELL_COVERED_CALL_AGGRESSIVE`** in its
   `_emit_to_intent` mapping. That emit code does not exist — plan §22.8's
   canonical 8 V1 emits only include `SELL_COVERED_CALL_PARTIAL`. Spec draft
   error; corrected in the appendix.
3. **`_project_collar_to_strike_selection` projects BOTH legs** (long put
   first, short call second). The spec's draft pseudocode projected only the
   short-call leg; the impl projects both because execution feasibility needs
   the long-put's liquidity too.
4. **`underlying_qty < 100` falls back to 100**, not raise. Spec proposed
   raising; shipped impl uses
   `max(positions.underlying_shares, _DEFAULT_COLLAR_QTY=100)` so dispatch
   survives recoverable input shapes (recommendation engine already gates
   `OPEN_COLLAR` on `has_long_stock`). Spec's open-question #3 is updated
   to reflect this.
5. **No JSON-fixture regeneration.** Spec proposed
   `scripts/regenerate_master_decision_fixtures.py --milestone m1.11b`; the
   script doesn't exist on `main` and the shipped test surface is inline
   Python fixtures in `test_decision_collar_dispatch.py` instead.

## Gotchas & non-obvious things

- **CHANGELOG was behind by two minor versions.** When CHANGELOG ownership
  lives in the same PR as the engine bump (per the `[1.4.0]` precedent in
  M1.13), a CHANGELOG-skipping PR silently drifts. The CI guard
  `scripts/check_engine_version_bump.sh` only enforces `engine/version.py`
  bumps — it does not enforce a corresponding `CHANGELOG.md` entry. Worth
  considering a `scripts/check_changelog_entry.sh` analog in a future cleanup:
  fail CI when `packages/engine/engine/version.py` changes in a diff but
  `CHANGELOG.md` does not gain a new top-level `## [x.y.z]` entry for the new
  version.

- **GitHub Actions are not enabled on the `knowlingo/option-mgmt-2026` fork.**
  Visiting `https://github.com/knowlingo/option-mgmt-2026/actions` and clicking
  "I understand my workflows, go ahead and enable them" is a one-time
  per-account UI action that has to happen before CI runs on PRs targeting
  this fork. The workflow file `.github/workflows/ci.yml` is present and
  correct (5 jobs: guards, api, engine, web, smoke); it simply will not fire
  until Actions is enabled. Upstream `csupenn/option-mgmt-2026` CI is healthy
  and green on every recent push.

- **`_DEFAULT_COLLAR_QTY = 100` fallback in `_dispatch_open_collar`** is a
  sharp edge for V2 — if a future ML node or rule change allows `OPEN_COLLAR`
  emits without `has_long_stock` gating, the dispatch will silently build a
  1-contract collar rather than raising. Acceptable for V1 (recommendation
  engine gates correctly), but worth flagging in any future audit of the
  rule-pipeline / dispatch boundary.

- **No CI verification possible from this fork.** Because Actions is disabled
  on `knowlingo/option-mgmt-2026`, the t03 doc-sync PR can't itself be CI-gated
  from here. The fork owner needs to enable Actions before the PR's checks
  will run. Once enabled, the doc-only diff should pass all 5 jobs trivially
  (no engine source touched, no Next.js / package.json touched, no broker
  imports introduced — all three guards short-circuit cleanly).
