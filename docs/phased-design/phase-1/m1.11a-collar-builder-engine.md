## Goal

Ship `packages/engine/engine/collar_builder/` — a pure-compute engine module that, given a chain snapshot and a user's profile + position, returns ranked `CollarStructure` candidates for one or more of the three intents (`zero_cost`, `income`, `defensive`) per master plan §9.10. **The module exists but is not yet wired into the Master Decision Engine** (that's M1.11b).

The module is the prerequisite for both M1.11b (Master Decision integration) and M1.16a (standalone `POST /engine/collar-builder` route), both of which are currently blocked.

### Public API — `engine.collar_builder.build()`

```python
def build(
    *,
    spot: Decimal,
    underlying_qty: int,
    chain: ChainSnapshot,
    profile: UserStrategyProfile,
    market_state: MarketStateResult,
    intents: list[CollarIntent] = list(CollarIntent),
    horizon_days: int | None = None,
    coverage_ratio: float | None = None,
) -> list[CollarStructure]:
    """
    Returns ranked candidate collars — typically one per requested intent.

    Raises:
        ValueError if underlying_qty < 100 (cannot collar fewer than 100 shares).
        ValueError if coverage_ratio * underlying_qty < 100 (would require 0 contracts).

    Returns empty list (not raise) when an intent has no feasible solution
    given chain liquidity and profile constraints — caller decides how to
    surface this. M1.11b's dispatcher treats an empty return as "no collar
    available; degrade to original recommendation action."
    """
```

`intents` defaults to all three (matches the `POST /engine/collar-builder` user-facing call surface). M1.11b's dispatcher will pass `intents=[ZERO_COST]` per §9.10 Integration.

### Dataclass shapes (per plan §7 `New: CollarBuilderRequest, CollarLeg, CollarStructure`)

```python
# packages/engine/engine/collar_builder/types.py — frozen dataclasses (ADR-0005)

class CollarIntent(str, Enum):
    ZERO_COST = "zero_cost"
    INCOME    = "income"
    DEFENSIVE = "defensive"

@dataclass(frozen=True)
class CollarLeg:
    kind: Literal["PUT", "CALL"]
    side: Literal["BUY", "SELL"]
    strike: Decimal
    expiry: date
    qty: int                     # contracts (not shares)
    delta: float                 # signed: call ∈ (0, 1], put ∈ [-1, 0)
    iv: float
    bid: Decimal
    ask: Decimal
    mid: Decimal
    premium: Decimal             # signed: + paid (debit leg), - received (credit leg)

@dataclass(frozen=True)
class CollarStructure:
    name: str                              # "Zero-cost 30d collar 405/430"
    intent: CollarIntent
    horizon_days: int
    long_put: CollarLeg
    short_call: CollarLeg
    net_debit_credit: Decimal              # negative = credit; per share, not per structure
    max_gain: Decimal                      # at expiry, per share (excluding stock cost basis)
    max_loss: Decimal                      # at expiry, per share
    upside_breakeven: Decimal              # spot price at which P&L = 0 going up
    downside_breakeven: Decimal            # spot price at which P&L = 0 going down
    capped_upside_pct: float               # (short_call.strike - spot) / spot
    protected_downside_pct: float          # (spot - long_put.strike) / spot
    confidence: float                      # M1.10 composer score, [0, 1]
    confidence_breakdown: ConfidenceBreakdown
    rationale: tuple[str, ...]             # short prose lines, ~3
    risks: tuple[str, ...]                 # short prose lines, ~3
    invalidation: tuple[str, ...]          # conditions that nullify the structure
    execution: Execution                   # M1.11 Execution Feasibility result for the combined legs
```

The engine ships frozen dataclasses; the API layer (M1.16a) projects to §7 Pydantic schemas. Field names match 1:1 (no transformation), consistent with the M1.13 pattern.

### Algorithm per intent (per plan §9.10)

**`zero_cost`** — solve for `(K_put, K_call)` such that `net_debit_credit ≈ 0` (within ±$0.05 per share), subject to:

- `protected_downside_pct ≥ profile.drawdown_tolerance`
- `short_call.delta ∈ profile.delta_target_band`
- both legs pass Execution Feasibility floors (`liquidity_score ≥ 0.5`, `fill_confidence ≥ 0.5` — calls into `engine.execution.assess()` per M1.11)
- `expiry` from `dte_band_days` (default `profile.dte_band_days[1]`)

**`income`** — solve for max `net_credit` subject to:

- `capped_upside_pct ≥ X` (config; default `4%`)
- `protected_downside_pct ≥ Y` (config; default `profile.drawdown_tolerance / 2`)
- same liquidity floors as zero-cost

**`defensive`** — solve for max `protected_downside_pct` subject to:

- `net_debit ≤ Z` (config; default `0.5%` of position notional)
- `short_call.delta ∈ profile.delta_target_band`
- same liquidity floors

### Solver implementation (`build.py` + `structures.py`)

1. **Filter chain to candidate expirations** within `[0, horizon_days]` days.
2. **Grid search** over `(K_put × K_call)` for each candidate expiry:
   - Calls: filter to `K_call > spot` (OTM); restrict to strikes where `delta ∈ profile.delta_target_band`.
   - Puts: filter to `K_put < spot` (OTM); restrict to strikes where `protected_downside_pct ≥ profile.drawdown_tolerance` (for zero_cost / defensive; for income, half the tolerance).
3. **Per-pair scoring**:
   - Compute `net_debit_credit = short_call.mid - long_put.mid` (signed, per share).
   - Compute `max_gain`, `max_loss`, breakevens, upside-cap %, downside-protection %.
   - Filter pairs failing Execution Feasibility floors (combined legs assessed via `engine.execution.assess()`).
4. **Per-intent objective**:
   - `zero_cost`: minimize `|net_debit_credit|`; tie-break by score.
   - `income`: maximize `net_credit` (i.e., minimize `net_debit_credit` since credit is negative); tie-break by score.
   - `defensive`: maximize `protected_downside_pct`; tie-break by score.
5. **Tie-break score** = `iv_score - event_score - illiquidity_penalty` (reuses M1.4a / M1.11 primitives).
6. **Return** the top candidate per intent. If no pair passes filters → empty list (caller decides).

### Confidence + rationale wiring

For each chosen `CollarStructure`, M1.11a calls `confidence.compose()` (M1.10) with the structure's component scores to produce `confidence` + `confidence_breakdown`. Rationale / risks / invalidation strings are intent-specific and live in `structures.py` as templates with f-string substitution from the structure's actual numbers (e.g., `"Protects {protected_downside_pct:.1%} downside in exchange for capping upside above ${short_call.strike}."`). Templates are inline — no new YAML file (keeps the M1.9 rule files unchanged).

## Open questions / risks

1. **Grid resolution.** §9.10 specifies "grid search over `(K_put × K_call)`". On a 45-strike × 4-expiry chain, that's 45×45×4 = 8,100 candidate pairs per intent, ×3 intents = 24,300 evaluations. Each evaluation is a sub-millisecond Execution Feasibility check (no I/O) so this stays well under the §7 latency budget (single-digit ms). If the chain grows (later phases), a pre-filter on `delta ∈ profile.delta_target_band` (calls) and `protected_downside_pct ≥ tolerance` (puts) cuts this to ~5×5×4 = 100 pairs. Plan §9.10 doesn't pin grid resolution — we go with full grid for V1 and document the pre-filter as a M3+ optimization.
2. **Zero-cost infeasibility on low-IV days.** When ATM IV is low (post-event-reprice, benign regimes), short-call premium often can't cover the long put within the profile's `delta_target_band`. The solver returns an empty list, M1.11b's dispatcher then falls back to the recommendation engine's secondary emit. This is the right behavior, but it means OPEN_COLLAR recommendations will sometimes degrade to NO_OP in production. Acceptable per the master plan; flagged here so the M1.19 ActionList UX (M1.19 spec) knows to handle "empty collar structure" gracefully.
3. **`coverage_ratio` semantics — partial coverage.** Plan §22.11 specifies `coverage_ratio` as a multiplier on `underlying_qty` to derive `contracts = floor(qty * ratio / 100)`. For a user with 150 shares and `coverage_ratio = 1.0`, this yields `contracts = 1` (covers 100 of 150 shares). The remaining 50 shares are explicitly uncovered. This is the only meaningful interpretation since options trade in 100-lot multiples, but it means `coverage_ratio = 0.5` on 150 shares yields `contracts = 0` (raises `ValueError`). The error message documents the math.
4. **`Execution` field on `CollarStructure` — single-pair or per-leg?** The §7 schema shows `execution: Execution` (singular) on `CollarStructure`. Combined-position execution feasibility is the right granularity for the UX, but `engine.execution.assess()` takes `actions: list[Action]`. We pass both legs as a 2-action list and the assessor aggregates. The `fill_confidence` on the combined `Execution` is the **minimum** of the two leg confidences (most-restrictive-leg semantics). Document this in `assess()`'s docstring.
5. **Master plan §9.10 says "1 candidate per intent" but the function signature returns `list[CollarStructure]` not a fixed-size tuple.** We honor the §9.10 promise ("typically 1 per requested intent") by returning at most `len(intents)` structures, with empty positions when an intent has no solution. Callers should check `len(result) ≤ len(intents)`; the order is intent-aligned to the input `intents` list. Documented in the `build()` docstring.
