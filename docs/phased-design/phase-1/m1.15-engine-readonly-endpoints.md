## Goal

Expose the three remaining engine sub-step entry points as read-only HTTP endpoints. `/engine/what-if` lets the UI preview a `DailyDecision` under hypothetical overrides without persisting. `/engine/market-state` and `/engine/flow-score` let the UI / Slack bot / scripts pull individual engine outputs without invoking the full pipeline.

## What ships

- **Endpoint** `POST /api/v1/engine/what-if` — runs `produce_daily_decision()` with caller-supplied `overrides` dict; returns a full `DailyDecision`. **Does not persist** (per §22.14). Identical request twice → identical response twice → zero rows written.
- **Endpoint** `POST /api/v1/engine/market-state` — runs `engine.market_state.classify()` only; returns a `MarketState` projection.
- **Endpoint** `POST /api/v1/engine/flow-score` — runs `engine.flow_score.compute()` only; returns a `FlowScore` projection (V1 contract).
- **New router** entries under `apps/api/app/routers/engine.py` (extends the existing M1.14 router; same `prefix="/engine"`, same auth deps).
- **New service functions** in `apps/api/app/services/decision_service.py`:
  - `run_what_if(*, session, user_id, ticker, as_of, overrides, inputs) -> DailyDecision`
  - `run_market_state(*, ticker, market_state_inputs) -> MarketStateResult`
  - `run_flow_score(*, ticker, chain_snapshot, spot, profile, expiry_focus) -> FlowScore`
- **New request schemas** in `apps/api/app/schemas/decision.py`: `WhatIfRequest`, `MarketStateRequest`, `FlowScoreRequest`.
- **Reuse**: existing `decision_to_jsonable_dict()` serializer (M1.14) for all three endpoint responses.

## Request / response schemas (Pydantic-shape pseudocode)

```python
# apps/api/app/schemas/decision.py — additions

class WhatIfRequest(BaseModel):
    """
    Identical body shape to DailyPlanRequest, plus an `overrides` dict
    applied to inputs before the engine runs. Per §22.14: NEVER persists.
    """
    ticker: str = "MSFT"
    as_of: datetime | None = None
    inputs: EngineInputs              # same hydrated bundle as DailyPlanRequest
    overrides: dict[str, JsonValue] = {}   # e.g. {"spot": 425.00, "iv_rank": 35}


class MarketStateRequest(BaseModel):
    """Inputs to engine.market_state.classify() per §22.3 extended signature."""
    ticker: str = "MSFT"
    # extended classify() inputs — all required, no service-layer hydration in V1
    spot: Decimal
    iv_rank: float
    iv_percentile: float
    hv_30: float
    expected_move_pct: float
    max_pain: Decimal
    pcr_volume: float
    pcr_oi: float
    days_to_next_event: int | None
    next_event_kind: str | None
    trend_strength: float                       # 0..1
    realized_vs_implied: float
    days_since_event: int | None
    # v1.2 §22.3 additions:
    days_to_nearest_opex: int | None
    iv_rank_change_1d: float | None
    gap_pct: float | None
    breakout_signal: float
    oi_concentration_at_max_pain: float


class FlowScoreRequest(BaseModel):
    """Inputs to engine.flow_score.compute() per §9.3a."""
    ticker: str = "MSFT"
    chain_snapshot: ChainSnapshotModel    # reuse the M1.14 Pydantic shell
    spot: Decimal
    expiry_focus: date                    # which expiry to score around
    profile: ProfilePayload               # full profile for decision tree (V1 contract)
```

Responses use the existing `decision_to_jsonable_dict()` for `WhatIf` and `MarketStateResult`; for `FlowScore`, return the V1 contract fields directly (the §9.3a-locked shape: `bullish_score`, `bearish_score`, `score`, `bias`, `pin_probability`, `gamma_risk`, `support_oi_wall`, `resistance_oi_wall`, `recommended_action`, `confidence`, `explanation`).

```python
# response envelopes — uniform with M1.14 style
class WhatIfResponse(BaseModel):
    decision: dict[str, Any]               # decision_to_jsonable_dict(DailyDecision)
    is_new_row: Literal[False] = False     # explicit: what-if never persists

class MarketStateResponse(BaseModel):
    market_state: dict[str, Any]           # decision_to_jsonable_dict(MarketStateResult)

class FlowScoreResponse(BaseModel):
    flow_score: dict[str, Any]
```

## DB migrations

**None.** All three endpoints are read-only. No tables added, no columns changed, no constraints touched.

The `inputs_hash` on `WhatIf` results still computes (the engine produces one); it is simply never persisted. This is identical to the M1.14 `persist=False` test path — already exercised by 16 unit tests.

## Open questions / risks

1. **Override key validation.** The `WhatIfRequest.overrides` dict is intentionally free-form per §7 (`overrides: dict`). Do we validate against a fixed key set, or pass-through and let the engine raise? **Default: pass-through with engine-side validation**, raise 422 if engine rejects. Revisit if confusion arises.
2. **`/market-state` and `/flow-score` body bulk.** These endpoints take fully-hydrated inputs. Until M1.17 (CSV upload) lands, callers must construct the snapshot themselves — same as M1.14 `daily-plan`. Once M1.17 ships, follow-up work can make these accept a lighter `{ticker, as_of}` body and hydrate from DB; tracked in M1.17 dev spec §"Knock-on effects".
3. **Idempotency on `/what-if`.** Two identical what-if calls produce identical responses but write nothing. This is the §22.14 contract. No ON CONFLICT path needed; verify in smoke that the COUNT(*) stays put.
4. **Profile dependency on `/flow-score`.** V1 `flow_score.compute()` reads `profile.style` for the decision-tree `recommended_action` (§9.3a). Until M1.17 ships profile-by-user_id lookup, callers pass `profile` in the request body. Acceptable for V1.
5. **Engine sub-step API symmetry with M1.16.** M1.16 will add `/engine/strike-candidates` and `/engine/execution-check` with similar request-body-heavy shapes. Pattern established here (Pydantic shell → `.to_engine()` → engine call → serializer) carries through.
