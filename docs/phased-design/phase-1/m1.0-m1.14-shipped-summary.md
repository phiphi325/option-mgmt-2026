## Engine version trajectory

The current `engine.__version__` on `main` is **`1.4.0`** (verified 2026-05-12 against `packages/engine/engine/version.py`). Trajectory from `0.x` to `1.4.0`:

| Version | Milestone | Reason |
|---|---|---|
| 0.x | M1.1 – M1.8 | pre-1.0 development |
| 1.0.0 | M1.9 | major bump — breaking signature change in `recommend()` (8 YAML rules contract) |
| 1.1.0 | M1.10 | minor — new Confidence Composer surface (`compose()` + `weights.yaml`) |
| 1.2.0 | M1.11 | minor — new Execution Feasibility surface (`assess()`, `liquidity_score`, etc.) |
| 1.3.0 | M1.12 | minor — new downgrade callback (`downgrade_if_needed()`) |
| 1.4.0 | M1.13 | minor — Master Decision Engine (`produce_daily_decision()`, `DailyDecision`) |
| 1.4.0 | M1.14 | unchanged — M1.14 changed `apps/api/`, not `packages/engine/engine/`; CI guard `check_engine_version_bump.sh` correctly skipped |
| 1.4.0 | M1.15 | unchanged — same reason as M1.14 (read-only API wrappers) |

The guard at `scripts/check_engine_version_bump.sh` (per ADR-0005) requires `engine.__version__` to bump whenever any file under `packages/engine/engine/` changes. None of M1.16 – M1.18 should bump it either (all three are API/UI work). The next planned bump is whichever milestone first lands `packages/engine/engine/collar_builder/` (currently expected to be M1.11a, blocked).

## What this milestone arc proves

- The 6-regime taxonomy + 5 scoring functions + Confidence Composer + Execution Feasibility produce coherent, deterministic `DailyDecision` outputs end-to-end. (The Collar Builder engine, M1.11a, is not yet shipped — see Patch v1.1 status table above.)
- The Master Decision Engine has been wired through to a real FastAPI endpoint with idempotent persistence to Postgres.
- CI's 5-job matrix (guards / api / engine / web / smoke) keeps drift, version bumps, and end-to-end behavior gated.
- The engine compiles and tests under Python 3.14 in CI; the conftest shims allow running under Python 3.9 sandboxes for local iteration.
