## Goal

When `recommendation_engine.recommend()` returns an action with emit code `OPEN_COLLAR`, the Master Decision Engine's `produce()` function now calls `engine.collar_builder.build(intents=[ZERO_COST])` and attaches the resulting `CollarStructure` to the `DailyDecision` payload. Today, M1.13's `produce()` falls through `strike_selector.select(intent="collar")` for these emits, which returns either empty or placeholder data because the Strike Selector has no collar-pairing logic.

M1.11b closes that gap. After this milestone, every `OPEN_COLLAR` emit in production produces a concrete two-leg structure with strikes, premiums, breakevens, and combined Greeks.

## What ships

### `engine/decision/produce.py` — dispatcher branch

The current `produce()` loop iterates `recommendation.actions` and calls `strike_selector.select()` for each. M1.11b adds an intent-dispatch fork:

```python
# engine/decision/produce.py — pseudocode of the modified loop

def produce(...) -> DailyDecision:
    recommendation = recommendation_engine.recommend(...)

    strike_selections: list[StrikeSelection] = []
    collar_structures: list[CollarStructure | None] = []

    for action in recommendation.actions:
        intent = _emit_to_intent(action.emit_code)   # new helper in produce.py

        if intent == "collar":
            structures = collar_builder.build(
                spot=spot,
                underlying_qty=underlying_qty,
                chain=chain,
                profile=profile,
                market_state=market_state,
                intents=[CollarIntent.ZERO_COST],     # §9.10: auto-call uses ZERO_COST only
                horizon_days=None,                    # use profile default
                coverage_ratio=None,                  # use profile default
            )
            if structures:
                structure = structures[0]
                # Project the short-call leg into a StrikeSelection so downstream
                # M1.11 Execution Feasibility + M1.12 downgrade still see a single
                # selection per action — preserves the equal-length-tuple invariant.
                strike_selections.append(_collar_leg_to_strike_selection(structure.short_call))
                collar_structures.append(structure)
            else:
                # No feasible collar at this snapshot. Degrade to NO_OP-style empty
                # placeholder; the recommendation's confidence will reflect the gap.
                strike_selections.append(_empty_strike_selection(action))
                collar_structures.append(None)
        else:
            selection = strike_selector.select(intent=intent, ...)
            strike_selections.append(selection)
            collar_structures.append(None)

    # ... existing M1.11 Execution Feasibility + M1.12 downgrade flow unchanged ...

    return DailyDecision(
        ...,
        strike_selections=tuple(strike_selections),
        collar_structures=tuple(collar_structures),     # new field
        ...,
    )
```

    strike_selections: tuple[StrikeSelection, ...]

    executions: tuple[Execution, ...]

Default of `()` preserves backward compatibility for any test fixtures or replay logs that pre-date M1.11b: they continue to construct `DailyDecision(...)` without the new field. The length-invariant is enforced by `produce()`, not by the dataclass.

## Request / response schemas

**Engine-layer only.** No Pydantic schema changes in M1.11b at the API layer — the `DailyDecision` Pydantic projection in `apps/api/app/schemas/decision.py` is permissive on the payload sub-shape (M1.13 design). The `collar_structures` field rides through the API's JSONB serializer transparently; clients parse it as part of the response body without an OpenAPI schema delta.

**However:** TS codegen runs as part of CI and the `packages/shared-types/` package will gain a `collar_structures` field on its `DailyDecisionResponse` type. This is the codegen drift gate's responsibility, and the regeneration must run in this PR to keep the front-end-side type accurate. (The M1.18 frontend ignores the field for now; M1.19+ consumes it.)

## DB migrations

**None.** The `daily_decisions` table is unchanged. The new `collar_structures` field lives inside the existing `payload JSONB` column — Postgres absorbs the shape change automatically. Replay queries (`SELECT payload FROM daily_decisions WHERE inputs_hash = ?`) continue to work; consumers that need the new field reach into the JSONB with `payload->'collar_structures'`.

The `engine_version` column on `daily_decisions` flips from `1.5.0` to `1.6.0` for rows produced after M1.11b ships. M1.16's `/engine/what-if` cache-hit logic uses `(engine_version, weights_version, inputs_hash)` as the cache key — the version bump correctly invalidates the cache for any pinned replay scenarios. Acceptable and expected.

## Open questions / risks

1. **Schema strategy — parallel field vs. nested in `StrikeSelection`.** We chose the parallel `collar_structures` field over extending `StrikeSelection` with an optional `secondary_leg`. Rationale: a `CollarStructure` carries pair-level data (`max_gain`, `max_loss`, breakevens, combined Greeks, `confidence_breakdown`) that does not fit naturally into a single-leg type. Parallel tuple keeps `StrikeSelection`'s existing contract intact, preserves the length invariant the rest of the pipeline relies on, and surfaces "this is a collar" cleanly to consumers (`collar_structures[i] is not None`).
2. **Auto-call vs. requirement-based call.** Master plan §9.10 says "When `market_state.regime ∈ {HIGH_IV_EVENT, POST_EVENT_REPRICE}` AND existing collar legs are absent, the Master Decision Engine calls `collar_builder.build(intents=[ZERO_COST])`." We implement this as: the recommendation engine already emits `OPEN_COLLAR` in those regimes per the M1.9 rule files; M1.11b's dispatcher reacts to the emit, not to the regime directly. This is the right separation of concerns (recommendation engine owns the "what to recommend" question; produce.py owns the "how to materialize the recommendation" question). It does mean that if a future M1.9 rule emits `OPEN_COLLAR` outside of `{HIGH_IV_EVENT, POST_EVENT_REPRICE}`, the dispatcher will still build a collar — which is correct behavior.
3. **`underlying_qty` plumbing in `produce()`.** M1.11a's `build()` requires `underlying_qty: int`. In M1.13, `produce()` already accepts `EngineInputs` which carries `positions: list[Position]`; the dispatcher sums `qty` across the `MSFT` positions to derive `underlying_qty`. If the user has zero MSFT shares, the recommendation engine should not have emitted `OPEN_COLLAR` in the first place (M1.9 rules check `has_long_stock`), but defensive code in the dispatcher raises a clear error if `underlying_qty < 100` reaches `build()`.
4. **Confidence composer interaction.** When `collar_builder` returns empty (no feasible structure), the dispatcher uses `_empty_strike_selection()` which carries `liquidity_score = 0` — this propagates through M1.11 Execution Feasibility to a low `fill_confidence`, which propagates through M1.10 Confidence Composer to a low overall `confidence`. The result is that the `DailyDecision` correctly reflects "we wanted to collar but couldn't" with low confidence + a degraded action. Frontends (M1.19) should treat low-confidence + null-collar as a "no actionable collar today" UX. Documented in M1.19's forthcoming dev spec.
5. **What if M1.11b ships before M1.16a?** That's the intended order (M1.11a → M1.11b → M1.16a). M1.16a's standalone `/engine/collar-builder` endpoint is independent — it doesn't depend on `produce()`'s dispatcher. The order is M1.11a → {M1.11b, M1.16a} (both can ship in parallel after M1.11a). Per the master plan §17 day-count (lines 2546–2548), the planned sequence is M1.11a → M1.11b → M1.16a + M1.16b, which is what we're following.
