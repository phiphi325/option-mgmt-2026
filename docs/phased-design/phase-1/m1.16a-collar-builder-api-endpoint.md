| Field | Value |
|---|---|
| **ID** | M1.16a |
| **Title** | Expose `engine.collar_builder.build()` as a standalone REST endpoint so users can query all three collar intents (ZERO_COST, INCOME, DEFENSIVE) directly, independent of the daily-plan pipeline |
| **Size** | S (0.5 day) |
| **Dependencies** | M1.11a (collar_builder engine module — required), M1.11b (OPEN_COLLAR dispatch wiring — informative, not blocking), M1.14 (`POST /engine/daily-plan` service patterns — required for service-layer conventions) |
| **Status** | planned |
| **Plan refs** | §7 (`CollarBuilderRequest` + `CollarStructure` schemas), §9.10 (Collar Builder integration — "When called via POST /engine/collar-builder, the user gets ALL requested intents ranked side-by-side"), §22.11 (H5 — `underlying_qty` resolved at service layer), Day 27.5 (Phase 1 Week 4 — M1.16a + M1.16b) |

## Goal

The engine's `collar_builder.build()` module was written in M1.11a and integrated into `produce_daily_decision()` in M1.11b. Today there is no way for a user to request collars for all three intents (ZERO_COST, INCOME, DEFENSIVE) side-by-side — the daily-plan pipeline always calls with `intents=[ZERO_COST]` only, and there is no standalone endpoint.

M1.16a adds `POST /engine/collar-builder` to `apps/api/app/routers/engine.py`. The user sends a `CollarBuilderRequest` and receives a ranked `CollarStructure[]` — typically three structures, one per intent — that they can use to select a collar and then route to execution.

This endpoint is referenced in the master plan API table (§7) and is the data source for any future "collar explorer" UI (post-Phase 1).

## What ships

### `apps/api/app/routers/engine.py` — new route

Add to the existing `router` (which already owns `/engine/daily-plan`, `/engine/recommend`, `/engine/what-if`, `/engine/execution-check`):

```python
@router.post(
    "/engine/collar-builder",
    response_model=list[CollarStructureResponse],
    summary="Build ranked collar candidates for up to 3 intents",
    tags=["engine"],
)
async def post_engine_collar_builder(
    request: CollarBuilderRequest,
    current_user: User = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
) -> list[CollarStructureResponse]:
    """
    Returns up to one `CollarStructure` per requested intent, ranked in
    intent-order (ZERO_COST → INCOME → DEFENSIVE).

    Resolves `underlying_qty` from the current user's MSFT position (§22.11 H5).
    Raises HTTP 422 if the user holds fewer than 100 MSFT shares.
    """
    return await collar_builder_service.build(
        request=request,
        user_id=current_user.id,
        db=db,
    )
```

### `apps/api/app/services/collar_builder_service.py` — new service

```python
# apps/api/app/services/collar_builder_service.py

from __future__ import annotations
from uuid import UUID
from fastapi import HTTPException
from sqlalchemy.ext.asyncio import AsyncSession

from engine.collar_builder import build as engine_build
from engine.collar_builder.types import CollarIntent, CollarStructure
from ..schemas.engine import CollarBuilderRequest, CollarStructureResponse
from ..db import queries as db_queries


async def build(
    *,
    request: CollarBuilderRequest,
    user_id: UUID,
    db: AsyncSession,
) -> list[CollarStructureResponse]:
    """Service layer for POST /engine/collar-builder.

    Per §22.11 H5: `underlying_qty` is resolved from the DB, never accepted
    from the request payload (removed from CollarBuilderRequest to prevent
    spoofing).
    """
    # 1. Resolve underlying_qty from positions
    qty = await db_queries.get_total_shares(
        db=db, user_id=user_id, ticker=request.ticker
    )
    if qty < 100:
        raise HTTPException(
            status_code=422,
            detail=f"Insufficient {request.ticker} shares (need >= 100, have {qty})",
        )

    # 2. Resolve profile, chain, market state, spot
    profile      = await db_queries.get_profile(db=db, user_id=user_id)
    chain        = await db_queries.get_latest_chain(db=db, ticker=request.ticker)
    market_state = await db_queries.get_latest_market_state(db=db, ticker=request.ticker)
    spot         = await db_queries.get_latest_spot(db=db, ticker=request.ticker)

    if chain is None:
        raise HTTPException(422, detail=f"chain not yet ingested for {request.ticker}")

    # 3. Apply request overrides (§7 CollarBuilderRequest)
    coverage = request.coverage_ratio or profile.max_coverage_pct
    horizon  = request.horizon_days   or profile.dte_band_days[1]

    # 4. Call engine — pure function, no I/O
    structures: list[CollarStructure] = engine_build(
        spot           = spot,
        underlying_qty = qty,
        chain          = chain,
        profile        = profile,
        market_state   = market_state,
        intents        = request.intents,
        horizon_days   = horizon,
        coverage_ratio = coverage,
        risk_free_rate = 0.05,
        dividend_yield = 0.0,
    )

    # 5. Project engine types → API response model
    return [CollarStructureResponse.from_engine(s) for s in structures]
```

## Request / response schemas

Per §7 canonical schemas, summarised above. Key constraints:

| Field | Rule |
|---|---|
| `ticker` | Validated: must be `"MSFT"` in Phase 1 (422 on others) |
| `intents` | Min 1, max 3. Duplicate intents collapsed to unique set. |
| `underlying_qty` | Not in request (§22.11 H5). Always resolved from DB. |
| Response order | Intent-order: ZERO_COST first, then INCOME, then DEFENSIVE — regardless of `intents` input order. Missing intents (no feasible collar) are omitted from the list, not filled with null. |

HTTP error codes:

| Code | Condition |
|---|---|
| 401 | Missing or expired JWT |
| 422 | `qty < 100` / chain not ingested / IV history < 30 rows |
| 500 | Unhandled engine exception (should never happen; engine is pure) |

## Open questions / risks

1. **Latency.** The grid search runs synchronously in the FastAPI async handler (CPU-bound). At Phase 1 chain sizes (< 100 contracts) this is fast (< 50ms). If in production the chain grows to 500+ contracts, offload to `asyncio.run_in_executor` (thread pool). Document this as a Phase 2 concern.

2. **Rate limiting.** This endpoint runs the engine three times (once per intent) and is more CPU-intensive than a simple DB read. Phase 1 has no rate limiting (internal tool). Phase 2 should add a per-user rate limit (e.g. 10 req/min) before exposing externally.

3. **Ticker validation.** Phase 1 only supports MSFT. The validator should be a config constant (`SUPPORTED_TICKERS = {"MSFT"}`) rather than a hardcoded string, so Phase 2 expansion (AAPL, GOOGL, etc.) is a one-line change.

4. **`from_engine()` projection.** The engine's `CollarStructure` is a frozen dataclass; the API schema is a Pydantic model. The projection must handle `Decimal` ↔ `float` precision and `tuple` ↔ `list` conversions. This is the only non-trivial part of the implementation — plan 30 min for this conversion.
