## Goal

After M1.21, the Today screen is complete. There is still no way to change the `UserStrategyProfile` that drives the engine (style, max_coverage, drawdown_tolerance, delta_target_band, etc.). Today these values default to the system defaults from the engine schema.

M1.22 creates the Settings page (`/settings`) with:

1. **`UserStrategyProfileForm`** — React Hook Form + zod, covering all 8 editable fields of `UserStrategyProfile`. Calls `PUT /profile` on save.
2. **`PersonaPresetButtons`** — three one-click preset buttons (Helen, Ravi, Diana) that fill the form with pre-configured values without saving. The user reviews the filled form and clicks Save.
3. **`AccountSection`** — email display, change-password link, and logout button (Phase 1 placeholder; full password-change flow is Phase 2).

After M1.22, the three personas defined in the master plan are usable from the UI, and the engine will reflect the saved profile on the next `/engine/daily-plan` call.

              <FormLabel>Max coverage — {(field.value * 100).toFixed(0)}%</FormLabel>

## Open questions / risks

1. **`UserStrategyProfile` tuple fields.** The Pydantic schema uses Python tuples for `delta_target_band: tuple[float, float]` and `dte_band_days: tuple[int, int]`. When serialised to JSON these become arrays. The `ProfileFormValues` zod type uses flat `delta_min` / `delta_max` / `dte_min` / `dte_max` fields for form compatibility, then `formValuesToProfile` reassembles them. Confirm the JSON serialisation of tuples at the API boundary before the PR opens.

2. **Single-handle Slider for ranges.** shadcn's `Slider` is single-handle in Phase 1. Using two separate form fields for min/max is a clean workaround that maintains zod cross-field validation. A dual-handle slider (e.g. `@radix-ui/react-slider` with `defaultValue={[min, max]}`) would be more intuitive but is a Phase 2 UX improvement.

3. **Profile versioning.** If the engine adds a new `UserStrategyProfile` field in Phase 2 (e.g. `preferred_collar_intent`), the form will not render it until explicitly added. The `extra="ignore"` Pydantic setting on `UserStrategyProfile` means new server-side fields are not rejected by the client `PUT` body — old form submissions remain valid. Document this as a drift concern to address in each profile-expanding milestone.

4. **Settings page SSR with auth.** The server component uses `next/headers` cookies for auth. This requires the API to accept the session cookie (not just JWT Bearer tokens). Confirm that the `GET /profile` endpoint supports cookie-based auth in the same way as the Today page's `daily-plan` fetch. If not, fall back to client-side fetch via React Query.

## Post-ship reconciliation (added 2026-06-10)

Shipped on `main` in [PR #17](https://github.com/knowlingo/option-mgmt-2026/pull/17)
(squash `a87fdfb`), **CI 5/5 green on the first push**. This milestone was
implemented **dependency-free (Path A)** and against the **real** engine
`UserStrategyProfile`, which diverges from the planning body above in several
concrete ways. **Where this appendix conflicts with the planning body above,
the appendix wins.**

### Why Path A (dependency-free)

The planning body builds the form on `react-hook-form` + `zod` +
`@tanstack/react-query` + shadcn `Form` / `Select` / `Slider` / `Tooltip` /
`toast` / `skeleton`. **None of these are installed** — `apps/web`'s only
runtime UI deps are `@radix-ui/react-dialog` + `recharts`, and `components/ui/`
contains only `button.tsx` + `dialog.tsx`. Adding them would require
regenerating the pnpm lockfile, which the CI `web` job consumes with
`pnpm install --frozen-lockfile` and which could not be regenerated in the
implementation environment. Rather than take five runtime dependencies for one
8-field form, M1.22 uses **native controlled inputs** (`<select>`,
`<input type="range">`, `<input type="checkbox">`) + React `useState`, and saves
via a **server action**. This matches the precedent set by M1.19–M1.21 (all
plain-Tailwind, shadcn-free).

### Deviations from the spec (verified against `engine/profiles.py` + `apps/api` `ProfileUpdateRequest`)

1. **Real 8-field schema (the spec's persona/zod fields do not exist).** The
   spec's `UserStrategyProfile` shape (`max_coverage`, `roll_aggressiveness`,
   `tax_sensitivity`, `iv_rank_sell_threshold`, `delta_target_band`,
   `dte_band_days`) is **not** the engine's. The real model is **8 fields**:
   `risk_tolerance` (`conservative`/`moderate`/`aggressive`), `income_need`
   (`low`/`medium`/`high`), `max_position_pct` `[0,1]`, `max_coverage_pct`
   `[0,1]`, `min_iv_rank_for_short_premium` int `[0,100]`,
   `prefer_collars_over_covered_calls` bool, `drawdown_tolerance` `[0,1]`,
   `style` (`income`/`balanced`/`growth`). The personas in `lib/personas.ts`
   and the `UserStrategyProfile` TS type in `lib/decision-types.ts` use these
   real fields; submitting the spec's fields would 422 against `PUT /profile`
   (`extra="forbid"`).
2. **Components live in `components/settings/`, not `components/today/`.** The
   spec's file map placed `UserStrategyProfileForm` / `PersonaPresetButtons`
   under `components/today/`; per plan §8/L4 (Settings screen) they belong under
   `components/settings/`. Tests are in `components/settings/__tests__/`.
3. **Mutation via a server action, not a React-Query client mutation.** The
   `access_token` cookie is **httpOnly** (read server-side in `lib/api/engine.ts`),
   so client JS cannot attach the bearer token — the save **must** round-trip
   through the server. `app/settings/actions.ts` exposes a `saveProfile` server
   action (the client→server boundary) that calls the server-only
   `lib/api/profile.ts::updateProfile`. It returns a `{ ok, error }`
   **discriminated union rather than throwing**, because errors thrown out of a
   server action are masked in production builds — a structured result is the
   only reliable way to surface a 422 detail or a 401 to the form.
4. **No client-side schema library (resolves open questions #1 + #2).** The 8
   fields are bounded by native `min`/`max`/`step` + fixed `<option>` sets, and
   the API's `ProfileUpdateRequest` (`extra="forbid"` + `Field` validators) is
   the authoritative validator. A zod mirror would be a second source of truth
   to drift, validating nothing the controls don't already constrain. The
   spec's cross-field rules (`delta_min < delta_max`, `dte_min < dte_max`)
   referenced **non-existent** fields; the real schema has **no** cross-field
   constraints. The "zod validates …" acceptance criterion is therefore **N/A
   by construction**. (Open question #1's tuple concern is moot — the real
   schema has no tuple fields. #2's single-handle-slider concern is moot — no
   range fields.)
5. **Auth pattern (resolves open question #4).** No new cookie-vs-bearer work:
   `getProfile` and the `saveProfile` action reuse the exact M1.18 pattern —
   read the JWT from the `access_token` cookie server-side, send it as an
   `Authorization: Bearer` header. `export const dynamic = "force-dynamic"` on
   `app/settings/page.tsx` keeps `/settings` out of build-time prerender, so
   `cookies()` is never called during `next build` (mirrors `app/today/page.tsx`).
   No React-Query fallback was needed.
6. **Native `title` tooltip; no `Suspense`/`Skeleton`.** `PersonaPresetButtons`
   uses the native `title` attribute for the persona description instead of a

   spec's `<Suspense>`/`<Skeleton>` wrapper is unnecessary. The `AccountSection`
   ships as an inline Phase-1 placeholder (`"Password change and account
   deletion available in a future update."`).
7. **Navigation.** Added a minimal top nav (Today · Settings) to
   `app/layout.tsx` as a server component with static `next/link`s (no
   active-state styling — Phase-1 minimal, per the spec's "exact nav chrome is
   implementation detail").
