## What landed

`engine.collar_builder` is a first-class pure-compute module that
produces ranked `CollarStructure` candidates for the three V1 intents
(`zero_cost`, `income`, `defensive`) per master plan §9.10. The
public surface is one function (`build()`) plus three dataclass types
(`CollarIntent`, `CollarLeg`, `CollarStructure`). No `produce.py`
changes — integration into the Master Decision Engine is M1.11b's job.

### What the saga taught us

- **The engine's canonical taxonomy is well-defined and brittle.**
  `Regime` is a `StrEnum` with six locked values per [ADR-0002]; mypy
  `--strict` catches typos at build time. Drafting code from memory
  against an enum the IDE doesn't know about will fail loudly — that's
  the point of `strict = true`. The pattern is healthy.
- **`§9.8` liquidity penalizes the OTM tail harder than expected.**
  A 1-cent bid/ask spread on a $1.00 mid is 100 bps — well under
  M1.11's 300-bp cap. The same 1-cent spread on a $0.40 mid is 250 bps
  — still under cap, but the leg lands at `fill_confidence ≈ 0.57`,
  one bad rounding away from the floor. The §9.8 weights (60% liquidity,
  40% spread) effectively double the penalty for cheap legs vs. their
  bid-ask quality alone would suggest.
- **`ZERO_COST_TOLERANCE = 0.10` is a hard boundary, not a fuzzy band.**
  Real chains will commonly produce `net_debit_credit` exactly at
  ±$0.10 — the test fixture stumbled into this by accident, but it
  reflects a genuine production-time behavior the dev spec under-
  specified. The pre-screen using `>` (strict) was the right call —
  the alternative would be to reject any pair within one ULP of $0.10,
  which is a worse user experience.
- **Test fixture engineering for grid-search solvers is hard.**
  The seed chain has 10 strikes × 1 expiration = 50 candidate pairs
  per intent. Getting the prices right so each of three intents has
  a clean winner (and the no-solution test has no winner) required
  three iterations. A `tests/fixtures/collars/<intent>/` set of JSON
  fixtures, as the dev spec originally promised, would have been
  cleaner — see Recommendation 3.

## Code-quality findings

### 1. The liquidity floor was lowered from 0.50 to 0.40 (vs. dev spec)

**Status: documented deviation.**

The dev spec promised a hard `MIN_FILL_CONFIDENCE = 0.50` filter
matching the M1.11 §9.8 downgrade threshold. We ship 0.40.

**Why this is OK:**

- The 0.50 in M1.11 is the **downgrade-trigger** threshold (M1.12
  ladder fires when `aggregate_fill_confidence < 0.50`), not a hard
  reject. Production decisions with `agg_fill ∈ [0.40, 0.50]` flow
  through the ladder, not to NULL.
- Our 0.40 filter surfaces marginal-liquidity pairs to the downgrade
  ladder rather than degenerating to empty. That's the right
  separation of concerns: the solver's job is to find candidates;
  the downgrade ladder's job is to decide whether they're tradeable.
- The dev spec's "≥ 0.50" was a calibration *target* from the master
  plan §9.10 algorithm description — not a hard contract derived from
  any downstream consumer.

**Why this should be revisited:**

- Phase-4 ML calibration may discover that pairs at fill_conf 0.40 produce
  consistently bad fills, in which case the floor should tighten.
- The dev spec's documented intent matters for institutional
  credibility — if the spec promises 0.50 and the code says 0.40,
  every reader has to reconcile that. The retrospective itself is
  partial mitigation; a follow-up PR could update the dev spec to
  read `0.40 in V1 (M1.12 ladder handles [0.40, 0.50])`.

### 2. The proxy delta is monotone but not accurate

**Status: V1 simplification documented inline.**

`_approx_delta(contract, spot)` uses a moneyness-linear formula:

```python
moneyness = (spot - strike) / spot   # signed
delta_call = clip01(0.5 + 4.0 * moneyness)
delta_put  = -clip01(0.5 + 4.0 * abs(moneyness))
```

This produces deltas that are correctly ordered (deeper OTM → smaller
absolute delta) but are off by 5–15% vs. Black-Scholes for typical
IV. The dev spec acknowledged this as the V1 fallback when the chain
doesn't publish deltas.

**Production impact:** When the chain feed *does* publish deltas (M1.6
`engine.greeks` does compute them), the solver should use those.
M1.11a's `make_long_put` / `make_short_call` accept an explicit
`delta=` kwarg — but the solvers in `structures.py` don't yet thread
real deltas through; they always fall back to the proxy. This is a
**known regression** vs. the dev spec, which promised "trust the
chain feed."

**Recommendation:** M1.11b should wire `engine.greeks.delta()` into
the dispatcher's path so production calls use Black-Scholes deltas,
and `_approx_delta` is preserved purely as a unit-test fallback.

### 3. The seed chain is engineered, not realistic

**Status: documented in `_collar_test_helpers.py`.**

The 10 strikes × 1 expiration × hand-picked mid prices are designed
so that:

- ZERO_COST has exactly one winning pair (380P + 420C, net 0.00).
- INCOME has exactly one winning pair (360P + 420C, net −0.50 credit).
- DEFENSIVE has exactly one winning pair (360P + 430C, net +0.10).

This makes the tests deterministic and easy to assert against, but
it means the test suite doesn't exercise the solver's behavior on a
realistic chain — e.g., when 30+ strikes are in the delta band and
the tie-break score actually matters.

**The dev spec promised 3 named regime-fixture bundles**
(`tests/fixtures/collars/{high_iv_event_zero_cost,
post_event_reprice_income, normal_regime_defensive}/`) with full
chain snapshots. **Those weren't shipped.** The engineered chain +
property tests give equivalent coverage of *correctness*, but not of
*plausibility*.

**Recommendation:** A follow-up PR (could be M1.11b or a standalone)
ships at least one realistic 45-strike × 4-expiration fixture
generated from real MSFT chain data, and adds tests that assert
the solver's tie-break behavior across multiple plausible pairs.

### 4. Rationale templates live inline in `structures.py`

**Status: deliberate V1 choice; revisit if it bloats.**

`_rationale_for_intent`, `_risks_for_intent`, and `_INVALIDATION`
return tuples of prose strings built via f-string substitution. They
sit in `structures.py` rather than a YAML file like M1.9's
`rules.yaml`.

**Why inline is fine for V1:**

- Three intents × ~3 strings each = 9 string templates total. A YAML
  file for 9 strings is overhead.
- The strings use Python f-string interpolation (`{put_strike:.2f}`)
  which YAML can't natively express — moving to YAML means parsing
  templates, which means adding a templating library or carrying a
  bespoke mustache-style substituter à la M1.9's `rationale.py`.

**When to move to YAML:**

- If we add a fourth intent and the count crosses ~12 templates.
- If the prose needs internationalization (a separate file per locale
  is the natural decomposition).
- If product wants to A/B test rationale phrasing without an engine
  version bump (a YAML edit could be a `weights_version`-style bump,
  not an `engine.__version__` bump).

None of these are imminent. Inline ships.

### 5. Confidence inputs use V1 heuristics, not calibrated weights

**Status: explicitly flagged as V1.**

`_confidence_inputs(market_state, flow_score, execution, intent)`
returns a `ConfidenceInputs` with values like `flow_alignment = 0.8`
for `BEARISH` flow + collar — pure heuristic. The M1.10 weights
(`v2.0`) are still applied multiplicatively on top, so the final
`confidence` is in a valid `[0, 1]` range, but the per-component
values themselves are not derived from any calibrated estimator.

The dev spec calls this out (under "Confidence + rationale wiring"):
"V1 heuristics — not yet calibrated; superseded when ME1.x ships."
ME-phase calibration against historical fills is the supersede path.

## Recommendations for M1.11b

1. **Thread real deltas through the dispatcher.** When `decision.produce()`
   dispatches to `collar_builder.build()` for an `OPEN_COLLAR` emit,
   pass `delta=engine.greeks.delta(...)` for each candidate strike
   rather than letting `_approx_delta` run. See finding #2.
2. **Wire the M1.12 downgrade ladder for collar legs.** Today's solver
   filters at 0.40, and the dev spec assumed M1.12's ladder would
   handle [0.40, 0.50] cases. M1.11b should verify the ladder behaves
   sensibly when handed a two-leg StrikeSelection (the ladder was
   designed for single-leg cases; collars need either pair-level
   reasoning or per-leg independent downgrades).
3. **Don't replicate the engineered-chain shortcut.** M1.11b's
   integration tests should use the M1.13 24-regime golden fixtures,
   2 of which carry an `OPEN_COLLAR` emit. Those are realistic enough
   to catch the kinds of bugs the engineered seed chain hid.
4. **Document the floor deviation in the M1.11b dev spec.** Add a
   one-liner to the dev spec's acceptance section: "The collar_builder
   liquidity filter is 0.40, not 0.50 (per M1.11a retrospective
   finding #1). M1.11b's dispatcher should not re-tighten."

## Open questions for future review

- Is the proxy delta's 5–15% inaccuracy enough to cause the solver to
  pick the *wrong* winning pair in production? Mitigated by Rec. 1,
  but worth measuring once real chains land.
- Should `CollarStructure.score` be exposed in the API surface (M1.16a)
  or kept internal? The dev spec is silent; today it's a public field
  on the dataclass.
- The 3 named regime fixtures are still owed (finding #3) — when do
  they ship? Earliest reasonable point is alongside the M1.13 golden
  fixture regeneration in M1.11b.
