## Goal

Close the decision→outcome learning loop in the UI. After M1.22 the Today and
Settings screens are live; there is still no way to record how a past
`DailyDecision` actually turned out. M1.23 adds `/outcomes` with:

1. **Manual entry** — a form that `POST`s an outcome tied to a `daily_decision_id`.
2. **History** — a cursor-paginated list of the user's outcomes (newest first) with "Load more".
3. **Edit** — inline `PATCH` of an existing outcome's fields.
4. **Stats** — at-a-glance counts by quality + an error-type histogram over the loaded rows.

This satisfies the Phase-1 done-bar item "manual outcome entry persists and
round-trips" and the §8 Outcomes-screen component intent.

## Reality review (planned scope vs. what shipped)

1. **No shadcn primitives.** Plan §8 "Shared primitives" lists Card / Badge /
   Slider / Tooltip / Form / Input / Select / Toast / Skeleton — but
   `apps/web/components/ui/` ships only `button.tsx` + `dialog.tsx` (the app's
   only UI deps are `@radix-ui/react-dialog` + `recharts`). As with M1.19–M1.22,
   this milestone is **dependency-free (Path A)**: native controlled inputs +
   plain Tailwind, no react-hook-form / zod / react-query.
2. **`pnl_*` are JSON strings, not numbers.** `OutcomeResponse.pnl_realized` /
   `pnl_unrealized` are Pydantic `Decimal`. This codebase serialises `Decimal`
   → JSON **string** (Pydantic v2 default; verified by the market smoke test's
   `float(body["spot"])` coercion vs. the profile test's bare `== 0.25` for a
   `float` field). The web wire type is therefore `string | null`, and the
   formatter coerces with `Number(...)`. (Contrast: the `DailyDecision` payload
   is `float` → JSON number.)
3. **No decision summary on the outcome.** `OutcomeResponse` carries only
   `daily_decision_id` (UUID) + the outcome fields. The plan's `OutcomeRow`
   wanted `as_of` / `regime` / `confidence` "decision summary" inline, but **no
   endpoint exposes that join**. The history row shows the decision **UUID**
   (monospace, truncated) + outcome fields. Surfacing decision metadata is an
   API follow-up (a `GET /outcomes` join or an `?expand=decision` param), out of
   scope for a web milestone.
4. **Manual entry needs a `daily_decision_id`.** `POST /outcomes` requires
   `daily_decision_id` + `horizon_days`. The form exposes an editable
   `daily_decision_id` field (paste from the Today card footer, which renders
   `decision_id`). A deep-link prefill (`/outcomes?decision_id=…` from a future
   Today "Record outcome" link) is deferred to keep the page-prop typing trivial.
5. **Stats are over loaded rows, not all-time.** `GET /outcomes` is
   cursor-paginated; computing all-time stats would require draining every page.
   `OutcomeStats` summarises the **currently-loaded** rows and is labelled as
   such. An aggregate stats endpoint is a Phase-2/3 concern (auto-fill lands in
   Phase 3 anyway).

### Wire types (`lib/outcome-types.ts`)

Mirror the shipped `app/schemas/outcome.py`. `Regime` is reused from
`option-mgmt-shared-types` (same source the decision types use). `pnl_*` are
`string | null` (see Reality review #2). `evaluated_at` / `id` /
`daily_decision_id` are ISO/UUID strings.

```typescript
export type OutcomeQuality = "good" | "neutral" | "bad";
export type OutcomeError =
  | "early_roll" | "late_roll" | "missed_breakout" | "over_coverage"
  | "under_coverage" | "wrong_strike" | "ignored_event" | "none";
export type OutcomeSource = "manual" | "auto";

export interface Outcome {
  readonly id: string;
  readonly daily_decision_id: string;
  readonly evaluated_at: string;
  readonly horizon_days: number;
  readonly pnl_realized: string | null;   // Decimal → JSON string
  readonly pnl_unrealized: string | null;
  readonly decision_quality: OutcomeQuality | null;
  readonly error_type: OutcomeError;       // non-null in responses (default "none")
  readonly actual_regime_realized: Regime | null;
  readonly regime_match: boolean | null;
  readonly notes: string | null;
  readonly source: OutcomeSource;
}
```

### Endpoints consumed (all from M1.17 — no new API)

| Method | Path | Notes |
|---|---|---|
| `GET /outcomes?since&limit&cursor` | list | cursor-paginated `(evaluated_at DESC, id DESC)`; `limit` 1–200 (default 50); 422 on bad cursor |
| `POST /outcomes` | create | 201; 404 (decision not owned), 409 (duplicate per decision), 422 |
| `PATCH /outcomes/{id}` | edit | 200; 404; 422 |

`source` is server-set (`manual`); not in the request body. Mutations go through
**server actions** (the `access_token` cookie is httpOnly, so the bearer can
only be attached server-side — same rationale as M1.22).

## Open questions / risks

1. **Decision summary join.** Showing `ticker` / `as_of` / `regime` / `confidence`
   next to each outcome needs an API change (`GET /outcomes` join or
   `?expand=decision`). Deferred; the row shows the decision UUID for now.
2. **Deep-link prefill.** `/outcomes?decision_id=…` from a Today-card "Record
   outcome" link is deferred (keeps the page prop signature trivial; the field
   is paste-editable meanwhile).
3. **All-time stats.** `OutcomeStats` is over loaded rows only (cursor
   pagination). An aggregate endpoint is Phase-2/3.
4. **`pnl_*` precision.** Values are passed to the API as the raw string the
   user typed (the server's `Decimal` is authoritative); the client only
   coerces with `Number(...)` for **display**, never for the request, to avoid
   float-rounding the stored value.

## Post-ship reconciliation (added 2026-06-10)

Shipped on `main` in [PR #19](https://github.com/knowlingo/option-mgmt-2026/pull/19)
(squash `9e55168`). The `web` CI job failed once on the first push — a **test
assertion bug**, not a code bug: `OutcomeRow.test.tsx` asserted the row showed
the outcome `id` when the row (correctly) renders the truncated
`daily_decision_id`; fixed in `88c7b0e`, then 5/5 green. This spec was authored
**against the shipped API**, so the body above already reflects reality; this
appendix records the as-built details + the two judgement calls made during
implementation. **Where this appendix conflicts with the body above, the
appendix wins.**

### As-built notes

1. **Shared `OutcomeFields`.** The create form (`OutcomeEntryForm`) and the
   row editor (`OutcomeRow`) render the **same** controlled `OutcomeFields`
   component, and both map through one pure module (`lib/outcome-form.ts`:
   `formValuesToCreateInput` / `formValuesToPatchInput`). This guarantees the
   create and edit paths can't drift on field set, `""`→`null` handling, or the
   `regime_match` tri-state. The decision-id field lives only on the create
   form (PATCH can't change it).
2. **Mutations are server actions** (`app/outcomes/actions.ts`:
   `createOutcomeAction` / `patchOutcomeAction` / `loadMoreOutcomesAction`),
   each returning a `{ ok, error }` union — same rationale as M1.22 (httpOnly
   `access_token` cookie; thrown server-action errors are masked in prod). The
   server-only `lib/api/outcomes.ts` reuses the `api/engine.ts` JWT-cookie
   pattern.
3. **`pnl_*` are JSON strings.** Confirmed the `Decimal`→string serialisation
   (Reality review #2); `lib/outcome-types.ts` types `pnl_realized` /
   `pnl_unrealized` as `string | null`, `lib/format.ts::formatPnl` renders
   signed USD / `"—"`, and the **write path sends the raw string** (never a
   `Number`).
4. **`OutcomeStats` over loaded rows.** Counts by quality + an error-type
   histogram, computed client-side from whatever has been loaded (initial SSR
   page + any "Load more" pages), captioned "over N loaded". Not an all-time
   aggregate (no endpoint for that).
5. **No `OutcomeFilters` component.** The plan §8 tree listed an
   `OutcomeFilters` slot (and `OutcomeRow` columns `as_of`/`regime`/`confidence`
   that don't exist on the API — Reality review #3). Shipped without a filter UI
   for Phase 1; the `GET /outcomes?since=…` param is wired in the client
   (`getOutcomes`) and a `since` filter is a low-cost follow-up. The history is
   newest-first with cursor "Load more".
