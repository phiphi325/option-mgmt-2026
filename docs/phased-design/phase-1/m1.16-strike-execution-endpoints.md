The reduced M1.16 ships **two endpoints**: `/engine/strike-candidates` + `/engine/execution-check`. After this PR merges, 7 of 8 engine sub-step endpoints in plan §7 are HTTP-reachable (the missing one being `/engine/collar-builder` — blocked on M1.11a).

## Goal

Ship the remaining strike + execution sub-step endpoints. After M1.16, every engine module already shipped through M1.13 is HTTP-reachable; the Today screen (M1.18) and what-if drilldowns can hit `/engine/strike-candidates` + `/engine/execution-check` directly without going through `/engine/daily-plan`.

## What ships

- **`POST /api/v1/engine/strike-candidates`** — calls `engine.strike_selector.select_strikes()` directly with a caller-supplied `Action` + `ChainSnapshot`. Returns the `StrikeSelection` JSON projection.
- **`POST /api/v1/engine/execution-check`** — calls `engine.execution.assess()` directly with a caller-supplied list of `StrikeLeg`s + optional `quantities`. Returns the `Execution` JSON projection.

### Dev-spec correction (the same kind of deviation as M1.15's flow_score)

The plan §9.4 prescribed signature is `select(intent, spot, chain_snapshot, profile, market_state, flow_score, ...)`. The engine actually ships `select_strikes(action, chain_snapshot, risk_free_rate, dividend_yield)` — i.e. takes an `Action` (from the M1.9 recommendation pipeline), not a high-level `intent`. The API layer accepts whatever the engine actually consumes (Action), following the same convention used for M1.15's `FlowScoreRequest` deviation. Same rationale: the engine is the source of truth; reshaping it to match a stale plan §9.4 signature would be code churn for zero functional benefit.

## Request / response schemas (Pydantic-shape pseudocode)

```python
# apps/api/app/schemas/decision.py — additions

class ActionPayload(BaseModel):
    """Pydantic projection of `engine.recommendation.Action` — the actual
    input to `engine.strike_selector.select_strikes()`."""
    emit: EmittedAction
    parameters: dict[str, float] = {}   # stable keys: target_dte, target_delta,
                                         #              size_pct, urgency_days


class StrikeCandidatesRequest(BaseModel):
    """Per §9.4 + the engine's real `select_strikes(action, chain_snapshot, ...)` signature."""
    ticker: str = "MSFT"
    action: ActionPayload
    chain_snapshot: ChainSnapshot
    risk_free_rate: float = 0.05
    dividend_yield: float = 0.0


class StrikeLegPayload(BaseModel):
    """Pydantic projection of `engine.strike_selector.StrikeLeg`."""
    contract: OptionContract        # already Pydantic in engine.types
    side: LegSide
    delta_target: float
    delta_actual: float
    delta_distance: float
    dte_actual: int
    mid_price: float | None = None


class ExecutionCheckRequest(BaseModel):
    """Per §9.8. Inputs match `engine.execution.assess(legs, quantities)`."""
    legs: list[StrikeLegPayload]
    quantities: list[int] | None = None    # defaults to [1] * len(legs)
```

Response envelopes reuse the M1.14 dict-of-jsonable pattern:
- `StrikeCandidatesResponse.strike_selection: dict[str, Any]` — the `StrikeSelection` JSON projection
- `ExecutionCheckResponse.execution: dict[str, Any]` — the `Execution` JSON projection

## Open questions / risks

1. **Action-vs-intent surface.** Same shape as M1.15's flow_score deviation — the dev spec adapts to the engine's real signature (`Action`-driven) rather than plan §9.4's `intent`-driven idea. Future PR can add an `IntentPayload` helper that constructs an `Action` from `intent + profile + market_state` if a higher-level API surface becomes useful for the UI.
2. **Profile hydration.** Both endpoints currently take all engine state in the body — `action.parameters` carries `target_delta`/`target_dte` (no profile needed). After M1.17 lands, a follow-up can add an "auto-derive action from latest market state + profile + emit code" path for the Today screen's drill-down flows.
3. **Empty-legs Execution semantics.** `assess()` accepts an empty legs tuple and returns an aggregate-only `Execution`. The endpoint accepts this and returns 200; documented in OpenAPI.
4. **Router size.** After M1.15 + this PR, `routers/engine.py` is ~330 lines. Still readable; revisit splitting (`routers/engine_decision.py` + `routers/engine_substeps.py`) only once M1.16a (collar-builder) and any future endpoints push it past ~600 lines.
5. **Collar-builder + market/latest re-entry.** Re-add to a future M1.16a dev spec once M1.11a (Collar Builder engine) lands; bundle market/latest into the M1.17 PR.
