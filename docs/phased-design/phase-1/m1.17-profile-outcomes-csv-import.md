## Scope revision (2026-05-12)

This dev spec originally listed the knock-on commitment (`DailyPlanRequest.inputs` becomes optional) as part of M1.17. During implementation that knock-on proved to be a genuinely distinct concern — modifying an existing endpoint, requiring a multi-step engine-call chain inside the API service to reproduce upstream computations (market_state + flow_score) from raw chain/iv rows. Keeping it inside the M1.17 PR would have pushed the diff past 2,000 lines.

|---|---|---|
| `/profile` GET/PUT | ✅ in this PR | — |
| `/outcomes` GET/POST/PATCH | ✅ in this PR | — |
| 5 × `POST /data/*/import-csv` | ✅ in this PR | — |
| `GET /market/{ticker}/latest` (M1.16b pickup) | ✅ in this PR | — |
| Migration `0003_imp_unique_constraints` (positions + option_positions UNIQUE) | ✅ in this PR | — |
| **Knock-on**: `DailyPlanRequest.inputs` optional + DB hydration | ✅ shipped in **M1.17.5 follow-up PR** | — |

The M1.17.5 follow-up adds `app/services/inputs_hydration_service.py` (chains `market_state.classify` + `flow_score.compute` from latest DB rows) plus modifies `DailyPlanRequest` (inputs now `EngineInputs | None`) and `produce_and_persist` to hydrate when None. 422 on `missing_positions` / `missing_chain` / `insufficient_iv_history`. Smoke-tested end-to-end (CSV uploads → daily-plan without inputs → 200 + persisted row + idempotency).

V1 hydration takes pragmatic shortcuts for `classify()` inputs that require historical context the schema doesn't yet track (breakout_signal=0.0, oi_concentration_at_max_pain=0.0, gap_pct=None, trend_strength=0.5). These are documented inline in `inputs_hydration_service.py` and earmarked for M2+ refinement.

## Goal

M1.17 is the milestone that turns the API from "synthetic-fixture-driven" into "user-data-driven". It ships three independent surface areas:

1. **`/profile`** — read/write the authenticated user's `UserStrategyProfile`. Persists to `users.strategy_profile` JSONB column (already created in M0.2 / §22.6).
2. **`/outcomes`** — manual outcome entry tied to a `daily_decisions` row; powers the §9.10 learning loop's Phase-1 manual path.
3. **`/data/*/import-csv`** — five CSV upload endpoints that populate `positions`, `option_positions`, `option_chain_snapshots`, `iv_history`, `events` from caller-supplied CSV per the §10 canonical formats.

### Unblocks (the headline impact)

After M1.17 ships, **`DailyPlanRequest.inputs` can become optional**. The M1.14 PR body's claim that "M1.15+ hydrates from DB" was inaccurate — it's M1.17 that lights up that path. Once a user has uploaded their positions + a fresh chain snapshot + 30+ days of IV history, the API service can hydrate `EngineInputs` from Postgres rows when the request omits `inputs`.

This is the **knock-on commitment** of M1.17: at the end of the same PR (or as a tiny follow-up), `DailyPlanRequest.inputs: EngineInputs | None = None` ships, with the `decision_service.produce_and_persist()` function checking for `None` and hydrating from DB. The same applies to `StrikeCandidatesRequest.chain_snapshot/market_state/flow_score/profile` and `CollarBuilderRequest`'s service-layer hydration completing per §22.11.

## What ships

### Profile

- **`GET /api/v1/profile`** — returns the authenticated user's `UserStrategyProfile`. If the user row's `strategy_profile` JSONB is empty `{}`, return defaults from `packages/engine/engine/profiles.py`.
- **`PUT /api/v1/profile`** — replaces the user's `strategy_profile`. Validates per §9.9 rules: `delta_target_band[0] < delta_target_band[1]`, `dte_band_days[0] < dte_band_days[1]`, `max_coverage ∈ [0,1]`, `iv_rank_sell_threshold ∈ [0,100]`. 422 on validation failure (RFC 7807 body listing the bad fields).

Note: §7 spells out `PUT`, not `POST`, for profile replacement. Use `PUT`. (Patch-style partial updates can be a future enhancement.)

### Outcomes

- **`GET /api/v1/outcomes?since=ISO&limit=N&cursor=…`** — paginated list of outcomes for the authenticated user. Cursor-based pagination over `(evaluated_at DESC, id DESC)`. Default `limit=50`, max `200`.
- **`POST /api/v1/outcomes`** — create an outcome tied to a `daily_decisions.id`. 422 if `daily_decision_id` doesn't belong to the authenticated user. 409 if outcome already exists (unique constraint on `daily_decision_id`).
- **`PATCH /api/v1/outcomes/{id}`** — update outcome fields. 422 on invalid field.

Outcomes table is owned by the authenticated user transitively via `daily_decisions.user_id`. Cross-user access returns 403.

### CSV Imports

Five endpoints, all `multipart/form-data`, all returning `{ inserted, updated, skipped, errors[] }`:

- **`POST /api/v1/data/positions/import-csv`** — upserts into `positions`. Idempotency: `UPSERT ON CONFLICT (user_id, ticker, opened_at)` — same user + ticker + opened_at means the same lot opening event.
- **`POST /api/v1/data/option-positions/import-csv`** — upserts into `option_positions`. Idempotency: `UPSERT ON CONFLICT (user_id, ticker, side, kind, strike, expiry, opened_at)`.
- **`POST /api/v1/data/chain/import-csv`** — inserts into `option_chain_snapshots`. **No unique constraint per §6 schema** — chain snapshots are append-only by design (multiple snapshots per `(ticker, fetched_at, expiry, strike, kind)` allowed if `fetched_at` differs). Duplicate-exact rows: skip.
- **`POST /api/v1/data/iv/import-csv`** — upserts into `iv_history` on `(ticker, ts)` (existing PK). Triggers post-insert validation per §22.12: `count(*) < 30 → reject` the upload as 422; `30 ≤ n < 60 → accept with warning`.
- **`POST /api/v1/data/events/import-csv`** — inserts into `events` per §10 canonical format. No native unique constraint; dedup by `(ticker, kind, scheduled_at, source)` at upload time.

## DB migrations

### Migration `0003_outcomes_dd_unique` (new)

`outcomes.daily_decision_id` is declared `UNIQUE` in §6 / 0001_init schema (the table-level `UNIQUE REFERENCES daily_decisions(id)` is already there). **No new constraint needed for outcomes idempotency.**

### Migration `0003_positions_user_ticker_opened_unique` (new — required for upsert idempotency)

Positions and option_positions need composite UNIQUE constraints to power `ON CONFLICT DO UPDATE` upserts:

```sql
-- positions
ALTER TABLE positions
  ADD CONSTRAINT positions_user_ticker_opened_unique
  UNIQUE (user_id, ticker, opened_at);

-- option_positions
ALTER TABLE option_positions
  ADD CONSTRAINT option_positions_user_leg_unique
  UNIQUE (user_id, ticker, side, kind, strike, expiry, opened_at);
```

Revision id: `0003_imp_unique_constraints` (24 chars — ≤32 budget per the M1.14 lesson).

### Migration `0004_chain_snapshot_natural_dedup` (optional)

Chain snapshots use append-only semantics — no UNIQUE needed. But a partial index for the `latest_chain` lookup in `MarketLatestSnapshot` (per §22.10) helps:

```sql
CREATE INDEX chain_latest_idx ON option_chain_snapshots(ticker, fetched_at DESC);
```

(Already exists in 0001_init as `chain_lookup_idx`. Verify and skip if duplicate.)

### Migration `0005_iv_history_validation_view` (skip in V1)

§22.12 validation logic lives in application code, not DB views. Defer.

**Net migrations for this PR: one (`0003_imp_unique_constraints`).**

## Open questions / risks

1. **CSV upload size limits.** §10 doesn't specify. Defaulting to FastAPI's 50 MB hard limit; recommend chunked uploads for production. For V1: cap at 10 MB per file at the router layer with an explicit 413.
2. **Cross-tenant data leakage on outcomes.** §15 (Security) requires per-user access checks. The `outcome_service.create_outcome()` MUST verify `daily_decisions.user_id == authenticated user_id` before inserting. Test `test_outcomes_403_cross_user` makes this a hard CI gate.
3. **Chain snapshot bloat.** `option_chain_snapshots` is append-only (§6 design). Repeat CSV uploads create new rows even for identical (ticker, fetched_at, expiry, strike, kind) tuples unless we dedup on exact-match. Default: dedup exact matches (skip) — caller can re-upload safely. Document this in the endpoint's OpenAPI description.
4. **`inputs` hydration determinism.** Two calls to `produce_and_persist()` with `inputs=None` on the same data **must** produce the same `inputs_hash` (otherwise ON CONFLICT idempotency breaks). The hydration query MUST be deterministic: order by `(fetched_at DESC, id)` with explicit tie-breakers. Add `test_hydration_deterministic_inputs_hash` to lock this in.
5. **Profile defaults source-of-truth.** `packages/engine/engine/profiles.py` defines defaults via the `UserStrategyProfile` Pydantic class. The API layer must NOT re-implement them. `profile_service.get_profile()` returns `UserStrategyProfile().model_dump()` when the JSONB is empty — single source of truth in `profiles.py`. Engine-version bump rules apply if the defaults change.
