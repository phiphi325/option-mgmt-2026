## Goal

Land the first user-visible page of the product: a working `/today` route in `apps/web` that fetches the authenticated user's `DailyDecision` from `POST /engine/daily-plan` and renders its headline structure. M1.18 ships the scaffolding (route, layout, data fetch) plus two of the seven Today-screen sub-components per §8 — the rest of the sub-components (M1.19 ActionList, M1.20 charts, M1.21 drawers) follow in subsequent milestones.

M1.18 is the **first Next.js milestone since M0.4** (which only shipped the auth/disclaimer shell). It establishes the conventions every subsequent UI milestone will follow.

## What ships

### Routing & data flow

- **`apps/web/app/today/page.tsx`** — Next.js App Router server component. Reads JWT from cookies, calls `POST /api/v1/engine/daily-plan` server-side (no `inputs` body — relies on M1.17's hydration path), passes the `DailyDecision` to a client component for rendering.
- **`apps/web/app/today/loading.tsx`** — Suspense fallback (skeleton with shadcn `Skeleton`).
- **`apps/web/app/today/error.tsx`** — error boundary. Renders the RFC 7807 error envelope from the API; surfaces "missing data" 422s with a CTA linking to the CSV upload page (which is M1.22 territory but stubbed here).
- **Redirect**: `apps/web/app/page.tsx` → `/today` (already in M0.4; verify and leave).

> **Post-ship note (yearline OM-Y3).** `app/today/page.tsx` now renders a second,
> **sibling** section below `<DailyDecisionCard>`: `<YearlinePanel>` (the read-only
> yearline evidence panel — card + headline distance-to-MA250 line). It is fed by a
> *separate, supplementary* server fetch (`getYearlineContext` → `GET
> /engine/yearline-context`) wrapped in try/catch so a yearline failure degrades to
> no-panel and never breaks the headline decision. It is **not** a child of
> `DailyDecisionCard` and is independent of the M1.19–M1.21 placeholder slots.
> See [ADR-0009](../../decisions/0009-adopt-yearline-statistical-context-provider.md)
> + [`docs/enhancements/yearline/implementation/`](../../enhancements/yearline/implementation/).

### Components

- **`apps/web/components/today/DailyDecisionCard.tsx`** — client component. Top-level container; accepts `decision: DailyDecision` prop. Renders the four immediate children from §8: `DecisionHeader`, `MarketStateBadge`, `StrategyTitle`, and **placeholder sections** for `ActionList` / charts / drawers (each renders a `Card` with a "Coming in M1.19/M1.20/M1.21" notice — explicit placeholder is better than a missing visual). This sets the layout grid for subsequent milestones to fill in.
- **`apps/web/components/today/DecisionHeader.tsx`** — `TickerSpotBlock` + `AsOfBadge` + `DataFreshnessBadge`. All three are content-only (no interactivity); render straight from `decision.ticker`, `decision.spot`, `decision.as_of`, `decision.data_freshness`.
- **`apps/web/components/today/MarketStateBadge.tsx`** — shadcn `Badge` with regime-specific color tokens per §8 theming ("regime-high-iv-event" amber, etc.). Six color variants in `apps/web/lib/regime-tokens.ts`.
- **`apps/web/components/today/StrategyTitle.tsx`** — renders `decision.recommended_strategy` (e.g. `"SELL_COVERED_CALL_PARTIAL"`) as a human-readable title with a verb-style label. Mapping table in `apps/web/lib/strategy-labels.ts`.
- **`apps/web/components/today/DisclaimerFooter.tsx`** — always-on footer per §15 disclaimer enforcement. Read once from `docs/disclaimers.md` content (compile-time inline; no runtime fetch).

### Typed API client

- **`apps/web/lib/api.ts`** — minimal fetch wrapper that adds the JWT header, parses RFC 7807 on non-200, and returns parsed `DailyDecision`. Uses the auto-generated types from `packages/shared-types/` (M0.6 codegen).
- **`apps/web/lib/api/engine.ts`** — `getDailyPlan({ ticker, as_of? }): Promise<DailyDecision>` server-only helper.

### Shared utilities

- **`apps/web/lib/regime-tokens.ts`** — maps `Regime` enum values to Tailwind utility class names. Single source of truth for regime → color in the UI.
- **`apps/web/lib/strategy-labels.ts`** — maps strategy verb codes to display strings. Initial set: `SELL_COVERED_CALL_PARTIAL`, `SELL_COVERED_CALL_AGGRESSIVE`, `BUY_PROTECTION`, `BUILD_COLLAR`, `REDUCE_COVERAGE`, `WAIT_AND_MONITOR`, `NO_OP`.

## Data fetch contract

```tsx
// apps/web/app/today/page.tsx — server component
import { getDailyPlan } from "@/lib/api/engine";
import { DailyDecisionCard } from "@/components/today/DailyDecisionCard";

export default async function TodayPage() {
  // Relies on M1.17 hydration — no `inputs` body needed.
  const decision = await getDailyPlan({ ticker: "MSFT" });
  return <DailyDecisionCard decision={decision} />;
}
```

The `getDailyPlan()` helper reads the JWT from cookies via `next/headers`. Errors propagate to `error.tsx`. If `error.tsx` receives an RFC 7807 with `type` matching `/missing-data` or `/insufficient-data`, it surfaces a CTA to the (eventually M1.22) upload page.

## Open questions / risks

1. **Server-component fetch vs. client RSC.** Next.js 16.2.6 App Router defaults to server components. The `/today` page is server-rendered for SEO-irrelevant content but to keep TTFB sharp. `DailyDecisionCard` is a client component because regime-token color application + subsequent milestones (charts, drawers) need interactivity. Boundary stays at the page → card transition; document with `"use client"` directive.
2. **No SWR / React Query in V1.** The Today screen is read-only at this milestone — no mutations, no revalidation. Defer React Query to M1.22 (Profile editor) where it becomes load-bearing.
3. **Fixture data for Playwright.** The `test_today_with_seeded_data` test relies on `scripts/seed_local.py` populating a test user with realistic CSV-shaped data. Coordinate with M1.17 — the seed script needs to gain a "Helen" persona seed mode. If M1.17 doesn't land that, M1.18 mocks the API in Playwright via `route.fulfill()`.
4. **Strategy label fallback.** Unknown strategy verb codes fall back to humanized snake-case. Acceptable for V1 (the engine may add new strategies; UI shouldn't crash). Track strategy whitelist in CI: a missing label triggers a console warning, not a test failure.
5. **Accessibility floor.** §8 requires color-blind safe rendering (confidence bar + labeled numerics). M1.18 doesn't render the confidence bar yet (that's M1.20), but the regime badge must include the regime name as text, not color only. Vitest assertion: badge contains regime string regardless of color class. WCAG AA color contrast for placeholders is not gated; charts (M1.20) will be.
