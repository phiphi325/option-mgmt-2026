# M1.24 — `expected.json` regeneration runbook

**Audience:** whoever picks up PR #3 (`feat/m1.24-master-decision-goldens`) to
finish the golden suite. The infrastructure, tooling, and all 12 fixtures'
`inputs.json` are merged on the branch and CI-green; the only remaining work is
**generating the 12 `expected.json` files** from the real engine and committing
them.

This cannot be done in the Hyperagent sandbox (Python 3.9, no `uv`). It must run
on a Python 3.14 dev environment with `uv` — the same toolchain CI uses.

---

## TL;DR

```bash
git fetch origin
git checkout feat/m1.24-master-decision-goldens
git pull

cd packages/engine
uv sync --dev
uv run python scripts/regenerate_decision_goldens.py --all

# Review every generated expected.json (see "Per-fixture verification" below).
git add tests/fixtures/master_decisions/*/expected.json
git commit -m "feat(M1.24): regenerate expected.json for all 12 fixtures"
git push
```

Then confirm CI is 5/5 green on the new commit, flip the PR from draft →
ready-for-review, and squash-merge.

---

## Why this step is manual

`expected.json` files are the **canonical output of `produce_daily_decision()`**
for each fixture's inputs. They MUST be generated by running the real engine —
hand-authoring them would defeat the entire purpose of a golden suite (a
hand-written "expected" value proves nothing; a recorded engine output that you
then diff against on every future change is what catches regressions).

The 3 example fixtures (01, 03, 11) currently carry **placeholder**
`expected.json` files containing the `_M1_24_PLACEHOLDER_RUN_REGEN_FIRST_`
marker; the other 9 have **no** `expected.json` yet. The replay harness
(`test_master_decision_goldens.py`) soft-skips both states with an actionable
message, which is why CI is green even though the goldens aren't populated.

## Step 1 — Regenerate

```bash
cd packages/engine
uv sync --dev
uv run python scripts/regenerate_decision_goldens.py --all
```

Expected console output (one line per fixture):

```
Regenerating 12 fixture(s)...
  ok: 01-high-iv-event-collar-feasible              ...b  regime=HIGH_IV_EVENT       emit=OPEN_COLLAR              confidence=0.xx
  ok: 02-high-iv-event-collar-no-feasible-pair      ...b  regime=HIGH_IV_EVENT       emit=OPEN_COLLAR              confidence=0.xx
  ...
```

The script writes each `tests/fixtures/master_decisions/<slug>/expected.json`
in place with sorted keys + 2-space indent + trailing newline.

If any fixture raises during `produce_daily_decision()` (e.g. a chain that
doesn't actually trigger the intended rule), the script prints a `SKIP` line
with the error. That fixture's `inputs.json` needs adjusting — see
"Troubleshooting" below.

## Step 3 — Idempotency self-check

The branch ships `test_regenerate_decision_goldens_idempotent.py`. After
committing the regenerated files, run it to confirm a second regen produces
byte-identical output (catches set-iteration / dict-ordering nondeterminism):

```bash
cd packages/engine
uv run pytest tests/test_regenerate_decision_goldens_idempotent.py -q
```

It also asserts the committed bytes match a fresh regen — so if it fails with
"regenerated bytes don't match committed expected.json", you have an
uncommitted regen drift; re-run `--all` and re-commit.

## Cross-environment determinism (the one thing worth double-checking)

The canonical serializer (`engine.decision.serialize_canonical`) rounds all
floats to 6 decimal places specifically to absorb platform ULP differences
between Apple Silicon (dev) and Linux x86_64 (CI). To be confident the goldens
are portable:

1. Regenerate on your local machine, commit, push.
2. Watch the `engine` CI job (runs on `ubuntu-latest` / x86_64). If the 12
   `test_golden_decision_matches[...]` cases pass there against goldens
   generated on your (likely ARM) Mac, determinism holds.
3. If any case fails on CI with a tiny float diff, the rounding precision needs
   loosening — bump `_FLOAT_PRECISION` in `serialize.py` from 6 to e.g. 4,
   regenerate, and document the change in the CHANGELOG. (Unlikely at 6 places,
   but this is the escape hatch.)

This is M1.24 dev-spec Open Question Q3; resolving it in practice is part of
finishing this PR.

## Troubleshooting

**A fixture's regen `SKIP`s with a `produce_daily_decision` error.**
The `inputs.json` shape is malformed or an enum value is wrong. Check:
- `Bias` / `RecommendedAction` use UPPERCASE values
  (`BEARISH`, `SELL_CALL_PARTIAL`, etc.)
- `ProfileStyle` / `RiskTolerance` / `IncomeNeed` use lowercase
  (`income`, `moderate`, `medium`)
- `Regime` / `OptionType` use UPPERCASE (`HIGH_IV_EVENT`, `CALL`)
- `as_of` is an ISO-8601 string ending in `Z`; chain `as_of` and contract
  `expiry` are `YYYY-MM-DD` date strings.

**A fixture regenerates but `matched_rule.id` is wrong (not the table above).**
The `inputs.json` doesn't actually trip the intended rule's predicates. Common
causes:
- A higher-priority rule in `rules.yaml` matched first (rules are first-match-wins).
  Adjust the inputs so only the intended rule's predicate set is satisfied.
- `iv_rank` thresholds: `high_iv_sell_call` needs `iv_rank ≥ 50`;
  `buy_long_dated_put_low_iv_trend` needs `iv_rank ≤ 30`.
- Regime allowlists: `high_iv_sell_call` only fires for
  `{HIGH_IV_EVENT, HIGH_IV_PIN, LOW_IV_RANGE}` (this is exactly why fixture 05
  was scope-refined from `LOW_IV_TREND` to `LOW_IV_RANGE`).
- Event windows: `open_collar_pre_event` needs `days_to_next_event ≤ 7`;
  keep non-collar fixtures' `days_to_next_event` well above 7.

**Fixture 12 doesn't escalate.** The chain isn't illiquid enough. Lower the OI
(`open_interest`) and `volume` further, or widen the bid/ask spread, until
every candidate leg's `fill_confidence` drops below the M1.12
`DOWNGRADE_THRESHOLD` (0.50) and the ladder exhausts.

**Fixture 02's collar is still feasible (non-null).** The chain is too liquid
for the degrade path. Lower OI/volume so the collar builder's 0.40 liquidity
floor filters every pair, OR widen the put/call premiums so no zero-cost pair
exists within ±$0.10.
